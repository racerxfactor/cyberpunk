Front-End Engineering

Real-time data that drives decisions.

Engineering a high-performance analytics dashboard that renders 2M+ data points at 60fps with live WebSocket streaming.

01 — Overview

From static reports to live intelligence.

The operations team was making million-dollar decisions based on PDF reports generated overnight. By the time they saw the data, it was already stale. They needed a real-time dashboard that could handle massive datasets without lag.

I led the front-end engineering effort — building a custom rendering pipeline, WebSocket data layer, and interaction framework from scratch.

02 — Problem

Performance at scale.

The technical challenges were substantial:

Volume

The dataset contained 2.4 million records updated every 5 seconds via WebSocket. Traditional charting libraries choked after 10K points. We needed a custom rendering approach.

Latency

Decision-makers needed sub-second visual updates when market conditions changed. The existing batch-processing pipeline introduced 4-hour delays.

Complexity

Users needed to cross-filter across 14 dimensions simultaneously while maintaining context. Every interaction had to feel instantaneous despite the data volume.

03 — Process

Building a custom rendering engine.

We evaluated every off-the-shelf solution and found none that met our performance requirements. So we built our own.

01

Benchmarking

Tested D3, Plotly, Chart.js, and deck.gl with our actual dataset. Identified the specific bottlenecks: DOM manipulation, layout thrashing, and GC pauses.

02

Canvas + WebGL Pipeline

Built a hybrid rendering pipeline — Canvas 2D for standard charts, WebGL for scatter plots and heatmaps exceeding 100K points. Implemented level-of-detail rendering.

03

Streaming Data Layer

Engineered a WebSocket middleware that diffs incoming data against local state, batches DOM updates into animation frames, and manages a circular buffer for historical queries.

04

Interaction Framework

Built cross-filter state management using a custom reactive store. Every filter operation runs in a Web Worker to keep the main thread free for rendering.

04 — Solution

60fps with 2 million points.

The final dashboard renders 2.4M data points with consistent 60fps interactions. Time-range brushing, dimensional cross-filtering, and drill-down all respond in under 16ms. Live data streams in via WebSocket with visual updates every 5 seconds.

The key engineering innovations: level-of-detail rendering that adapts chart fidelity to zoom level, a Web Worker pipeline for cross-filter calculations, and a custom virtual scrolling implementation for the data table that handles 100K+ rows.

What I learned

The most important optimization wasn't algorithmic — it was perceptual. We discovered that users couldn't perceive detail beyond ~50K points at any zoom level. Level-of-detail rendering let us show a visually identical chart with 1/50th the data, then progressively refine as users zoomed in.
