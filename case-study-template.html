01 — Overview

A fragmented ecosystem needed one source of truth.

The company had grown through acquisitions, leaving 12 product teams each building UI with their own patterns, tokens, and component libraries. Inconsistency was everywhere — buttons looked different across products, spacing was unpredictable, and every new feature required rebuilding base components from scratch.

I was brought in to design and engineer a unified system that would serve as the single source of truth for the entire product organization.

02 — Problem

What we were up against.

Through stakeholder interviews, design audits, and codebase analysis, three core problems emerged:

Design fragmentation

A visual audit surfaced 23 different button styles, 14 color palettes, and 8 type scales across the product suite. Users experienced a jarring shift every time they navigated between tools.

Engineering inefficiency

Without shared components, engineers rebuilt common UI patterns for every project. An average feature took 40% longer than it should because foundational elements weren't reusable.

Broken handoff

Designers and developers spoke different languages. Figma designs didn't map to code. Redlines were ignored. Specs were misinterpreted. The gap between intent and implementation was massive.

03 — Process

From audit to architecture.

The project moved through four distinct phases, each building on the last. Here's how we went from chaos to cohesion.

01

Discovery & Audit

Catalogued every component, pattern, and token across all 12 products. Interviewed designers, engineers, and PMs to understand pain points and workflows.

02

Token Foundation

Defined a semantic token layer — color, spacing, typography, elevation, motion — that could flex across brands while maintaining consistency.

03

Component Architecture

Designed and built 48 core components in Figma and React simultaneously, ensuring 1:1 fidelity between design and code from day one.

04

Adoption & Governance

Created onboarding guides, Storybook documentation, and a contribution model. Ran weekly office hours and embedded with pilot teams.

04 — Solution

A living system, not a static library.

The final system includes 48 production-ready components with full Figma ↔ React parity, a semantic token architecture supporting light/dark/high-contrast modes, automated visual regression testing in CI, and interactive Storybook documentation with live code editing.

The key innovation was building the design-to-code pipeline so that a designer's changes in Figma could trigger token updates in the codebase — closing the gap between intent and implementation permanently.

What I learned

The biggest lesson wasn't technical — it was organizational. A design system succeeds or fails based on adoption, and adoption is a people problem. The weekly office hours, embedded support, and contribution model were just as important as the component quality itself.
